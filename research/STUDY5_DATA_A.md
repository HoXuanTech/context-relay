## Summary1 — T1 Test Context (after S1)
*Input: S1 (~6,800 words) → compressed to ~1,000 words*

This is a personal recipe collection web app being built from scratch in plain HTML, CSS, and vanilla JavaScript — no framework, no build tooling. The data source chosen is TheMealDB, a free public recipe API that requires no API key and allows direct browser fetch calls because its responses include permissive CORS headers. The decision was made after evaluating alternatives: some required paid plans for reasonable rate limits, others needed server-side proxies to avoid CORS restrictions, and some returned ingredient data as a combined string rather than separate name and measure fields. TheMealDB stood out for combining free access, browser-compatible CORS, and rich per-recipe data: title, thumbnail image URL, category (e.g., Seafood), area (e.g., Japanese), ingredient names with their corresponding measures, a YouTube video link, and a source page URL.

The API surface in active use covers four endpoints, all under the base URL `https://www.themealdb.com/api/json/v1/1/`. Text search uses `search.php?s={query}` — submitting "chicken" returns an array of meals whose titles contain the word. Fetching a single recipe by ID uses `lookup.php?i={id}` where the ID is a numeric string assigned by TheMealDB (e.g., `52772` for Teriyaki Chicken Casserole). The categories list uses `categories.php`, returning all available meal categories. The areas (cuisines) list uses `list.php?a=list`, returning all available cuisine names. Categories and areas are used to populate the filter chip UI planned for Session 4.

TheMealDB's raw meal object is inconvenient for the rest of the app because it spreads ingredient data across twenty parallel fields: `strIngredient1` through `strIngredient20` and `strMeasure1` through `strMeasure20`, all at the top level of the object alongside the title, thumbnail, category, and other properties. Unused ingredient slots contain empty strings rather than being absent. A dedicated `mapper.js` file normalizes every raw TheMealDB response into a consistent schema before any other part of the app sees the data. The unified Recipe schema is: `{id, title, thumbnail, category, area, tags, ingredients, youtubeUrl, sourceUrl}`. The `ingredients` field is an array of objects with two string fields each: `{name, measure}`. Storing ingredients as objects rather than concatenated strings like `"1/2 cup flour"` was a forward-looking decision — a serving scaler planned for Session 3 will need to extract the numeric portion of the measure separately from the unit text, which is only feasible if name and measure are already in separate fields.

Recipe cards display title, thumbnail, category, and area. Images are fetched from the TheMealDB CDN. Some thumbnail URLs in the dataset return 404 responses or resolve to broken images. An `onerror` handler on each card's `<img>` element catches load failures and replaces the failed URL with `PLACEHOLDER_IMAGE`, a constant defined in `constants.js` pointing to a gray placeholder image. Defining the fallback in constants.js means the URL can be updated in one place rather than scattered through the rendering code.

The JavaScript is organized across five files with a deliberate load order that avoids circular dependencies: `constants.js` (shared constants and STORAGE_KEYS), `mapper.js` (TheMealDB raw-to-schema normalization), `api.js` (fetch wrappers for all TheMealDB endpoints), `ui.js` (DOM manipulation utilities, card rendering, status messages), `app.js` (event wiring and application initialization). Files are included via `<script>` tags at the bottom of `index.html` body in this sequence. Each file only references names defined in earlier files. No module bundler or `import/export` syntax is used; global scope sharing between files is the intentional coordination mechanism.

Safety Rule #1 was locked in at the end of S1: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`. This localStorage key string must never be changed regardless of refactoring, renaming, or tidying. Changing it would silently orphan any favorites a user had saved under the old key — the data would persist in localStorage under the original key with no migration path and no error visible to the user. The `STORAGE_KEYS` object in `constants.js` is the single source of truth for all localStorage key strings used by the app.

Favorites are stored as a JSON-serialized array of recipe ID strings (e.g., `["52772", "52804", "52859"]`). Full Recipe objects are not stored — only the IDs. This decision was established in S1 and reinforced as Safety Rule #2 in S2. At the end of S1, the search UI is functional: querying returns recipe cards, cards display correctly with onerror fallback for broken images. The favorites heart icon is visible on each card but the toggle logic and the favorites view are not yet wired — those are the S2 tasks.

## Summary2 — T2 Test Context (after S2)
*Input: Summary1 + S2 (~7,800 words) → compressed to ~1,000 words*

A personal recipe collection web app, plain HTML/CSS/vanilla JavaScript, no framework, no build tooling. Data source is TheMealDB (free, no API key, CORS-open). Four endpoints in use: `search.php?s=` for text search, `lookup.php?i=` for single-recipe fetch by numeric ID, `categories.php` for category list, `list.php?a=list` for areas list. All under `https://www.themealdb.com/api/json/v1/1/`. Raw TheMealDB meals are normalized via `mapper.js` into the unified schema `{id, title, thumbnail, category, area, tags, ingredients [{name, measure}], youtubeUrl, sourceUrl}` before the rest of the app sees them. Ingredients use the object shape because the S3 serving scaler needs separate name and measure fields. Broken thumbnail images fall back to `PLACEHOLDER_IMAGE` via `onerror`. File load order: `constants.js` → `mapper.js` → `api.js` → `ui.js` → `favorites.js` → `app.js`. Safety Rule #1 is in force: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`, locked in constants.js, never change.

The favorites system is now complete. Safety Rule #2 was established in S2: favorites must be stored as a JSON array of recipe ID strings only. Full Recipe objects must never be stored under the favorites key. The reasoning has two parts: the localStorage origin limit is 5MB, and a typical Recipe object with an ingredient array, thumbnail URL, area, category, and tags can be several kilobytes — storing full objects for dozens of favorited recipes would approach the limit, whereas storing IDs costs a few bytes each. The second reason is architectural clarity: the favorites key represents "which recipes the user cares about" as an index, separate from "what those recipes contain," which lives in TheMealDB's servers. The accepted consequence is that loading the favorites view requires one `lookup.php?i=` network call per favorited ID to fetch the full recipe data before rendering. This is acceptable because the favorites list for a personal use case is expected to be small (tens of recipes, not hundreds).

The `toggleFavorite(id)` function is the central write operation for the favorites system. It reads the current ID array from localStorage using `JSON.parse`, adds the ID if it is absent or removes it if it is present, then writes the updated array back to localStorage using `JSON.stringify`. The heart icon on the clicked card is updated to reflect the new state. All localStorage reads use `JSON.parse` wrapped in `try/catch` — if the stored value is corrupted or has been manually edited to invalid JSON, the catch block returns an empty array as the default rather than propagating an exception. This defensive read pattern applies to every localStorage key in the app.

`loadFavorites` is called when the user switches to the Favorites tab. It reads the ID array from localStorage and issues one `lookup.php?i=` call per ID, collects the resolved Recipe objects, and renders them into `favoritesGrid`. The data shown in the favorites view always reflects TheMealDB's current records for those IDs — if TheMealDB updates a recipe's thumbnail or title, the favorites view will show the updated version on next load.

The UI uses two separate grid containers in the DOM: `searchGrid` for search results and `favoritesGrid` for saved recipes. Both are present in the HTML at all times. Switching between the Search tab and the Favorites tab shows one grid and hides the other using CSS `display` toggling. This means search results are preserved when the user switches to Favorites and back — no re-fetch is triggered by tab switching. A third tab for custom recipes is planned for S5 but not yet present.

`favorites.js` was added to the script load order between `ui.js` and `app.js`. It defines `toggleFavorite`, `loadFavorites`, and helper functions for reading and writing the favorites array. All functions in `favorites.js` reference only names defined in `constants.js`, `mapper.js`, `api.js`, or `ui.js`, maintaining the load-order contract.

At the end of S2: search with result cards, favorites toggle on each card (heart icon, localStorage read/write), favorites view rendering fetched recipe data, two-tab navigation. Not yet built: the recipe detail modal (full ingredient list, serving scaler, YouTube and source links) — that is S3. The compound filter and search history are S4.

## Summary3 — T3 Test Context (after S3)
*Input: Summary2 + S3 (~7,800 words) → compressed to ~1,000 words*

A personal recipe collection web app, plain HTML/CSS/vanilla JavaScript, no framework, no build tooling. TheMealDB API (free, no key, CORS-open). Base URL `https://www.themealdb.com/api/json/v1/1/`. Endpoints: `search.php?s=`, `lookup.php?i=`, `categories.php`, `list.php?a=list`. Unified Recipe schema: `{id, title, thumbnail, category, area, tags, ingredients [{name, measure}], youtubeUrl, sourceUrl}`. Broken thumbnails use `PLACEHOLDER_IMAGE` via onerror. File load order: constants → mapper → api → ui → favorites → app. Safety Rule #1: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`, never change. Safety Rule #2: favorites stored as ID string array only — `loadFavorites` fetches full data with one `lookup.php?i=` call per ID. All localStorage reads are wrapped in `try/catch` with empty array as default. `toggleFavorite` reads, modifies, and writes the ID array. Two-tab navigation; `searchGrid` and `favoritesGrid` both persist in DOM.

The recipe detail modal was built in S3. Clicking a recipe card opens an overlay that displays the full recipe: title, thumbnail image, category and area labels, the complete ingredient list with measures, a serving scaler with +/- buttons, a YouTube embed link, and a source page link.

The native HTML `<dialog>` element was explicitly evaluated and rejected. The reasons: `::backdrop` pseudo-element styling is inconsistent between Chrome and Firefox (Chrome clips it to the viewport, Firefox does not), the `returnFocus` behavior on dialog close differs between implementations (Chrome focuses the element that opened the dialog; Firefox's behavior is less predictable without explicit management), and polyfilling `<dialog>` for browsers that don't support it natively adds ongoing maintenance cost. A `<div>` overlay with `role="dialog"`, `aria-modal="true"`, and `aria-labelledby` pointing to the modal's heading element was used instead, with fully manual focus management. This approach is now the required pattern for all overlay UI in the project — any future modal-like component must use the div overlay approach, not the native element.

Focus trapping is implemented via a `keydown` listener on the modal container. When Tab is pressed, the listener identifies all focusable elements within the modal (buttons, links, inputs), and if focus would move beyond the last focusable element it wraps to the first; Shift+Tab wraps in the reverse direction. The modal opens by immediately calling `.focus()` on the close button, placing keyboard focus inside the modal from the moment it appears. On close — whether triggered by clicking the X button, pressing Escape, or clicking the backdrop overlay outside the modal content area — focus returns to the recipe card element that was clicked to open the modal. The card's element reference is captured and stored before the modal opens.

The serving scaler operates on the `ingredients` array of the Recipe object. The default serving count is 4. Pressing + increments the count; pressing - decrements it with a minimum of 1. Each time the count changes, the displayed ingredient quantities are recalculated. `parseQuantity` handles three cases: plain integers and decimals (e.g., `"2"` → 2); fractions (e.g., `"1/2"` → 0.5, by splitting on `/` and dividing the two parts); and non-numeric strings (e.g., `"to taste"`, `"as needed"` → returned as-is, not scaled). Scaled numeric quantities are passed through `Math.round(value * 100) / 100` to eliminate floating-point rounding artifacts — without this, scaling `"1/3 cup"` by 3 would yield `0.9999999999999999` instead of `1`.

At the end of S3: search with recipe cards, favorites toggle and favorites view, recipe detail modal with full ingredient list, serving scaler (parseQuantity, fraction handling, float rounding), focus trapping, Escape key dismissal, backdrop click dismissal, return-focus on close, YouTube and source links. Not yet built: compound filter (category + area), search history — those are S4. Custom recipe CRUD is S5.

## Summary4 — T4 Test Context (after S4)
*Input: Summary3 + S4 (~7,800 words) → compressed to ~1,000 words*

A personal recipe collection web app, plain HTML/CSS/vanilla JavaScript, no framework, no build tooling. TheMealDB API (free, no key, CORS-open). Endpoints: `search.php?s=`, `lookup.php?i=`, `categories.php`, `list.php?a=list`. Recipe schema: `{id, title, thumbnail, category, area, tags, ingredients [{name, measure}], youtubeUrl, sourceUrl}`. File load order: constants → mapper → api → ui → favorites → app. Safety Rule #1: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`, never change. Safety Rule #2: favorites as ID strings only; `loadFavorites` fetches full data per ID via `lookup.php?i=`. `try/catch` on all localStorage reads; default to empty array. Modal: `<div>` overlay with `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, manual focus trapping, Escape + backdrop dismissal, return-focus to triggering card. `parseQuantity` handles fractions and uses `Math.round(v * 100) / 100` for float safety. Serving scaler default is 4.

The compound filter was built in S4. Category names are fetched from `categories.php` and rendered as a row of clickable chip buttons. Area (cuisine) names are fetched from `list.php?a=list` and rendered in a separate row. Both lists are fetched on app load. Filter state is tracked in a `currentFilters` object with two fields: `{category: string|null, area: string|null}`. When a chip is clicked, its value is set on the corresponding field; clicking an already-active chip resets the field to null (deselects). The filter applies AND logic: a recipe card must match both the selected category and the selected area to remain visible. If only one filter is active, only that constraint applies. Recipes that do not match the active filter combination are hidden with `display: none` rather than removed from the DOM — this preserves the card elements and their event listeners without requiring re-fetching from the API when filters are changed.

Roving tabindex is used for keyboard navigation within each filter chip group. At any time, exactly one chip in each row has `tabindex="0"` and all others have `tabindex="-1"`. Pressing the right arrow key moves focus to the next chip and transfers `tabindex="0"` to it. Left arrow moves in the reverse direction. Tab exits the chip group entirely and moves focus to the next focusable element outside the group. This pattern lets keyboard users navigate filter chips efficiently without pressing Tab for every chip. The implementation maintains a `currentIndex` per group and updates it on arrow key events.

Search history is stored in localStorage under `STORAGE_KEYS.SEARCH_HISTORY = 'recipe-search-history'`. This key is locked in under the same "never change" principle as the favorites key — renaming it would silently orphan any stored history. The history is a JSON array of query strings capped at 10 entries. When a new query is added, it is deduplicated (an identical query already in the list is removed before prepending the new one, preventing exact duplicates). When the list exceeds 10 entries, the oldest (last) entry is evicted. When the search input receives focus, the stored history is displayed as clickable chips beneath the input. Clicking a history chip populates the input field and fires an immediate search without requiring the user to press Enter. `STORAGE_KEYS.SEARCH_HISTORY` is defined in `constants.js`.

At the end of S4: search with results, favorites toggle and view, recipe detail modal with serving scaler and focus trapping, compound category+area filter (AND logic) with roving tabindex, search history (localStorage, max 10, deduplicated, auto-populated on input focus). Custom recipe add/edit/delete is not yet built — that is S5. Offline cache, performance, and deployment are S6.

## Summary5 — T5 Test Context (after S5)
*Input: Summary4 + S5 (~7,800 words) → compressed to ~1,000 words*

A personal recipe collection web app, plain HTML/CSS/vanilla JavaScript, no framework, no build tooling. TheMealDB API (free, no key, CORS-open). Recipe schema: `{id, title, thumbnail, category, area, tags, ingredients [{name, measure}], youtubeUrl, sourceUrl}`. File load order: constants → mapper → api → ui → favorites → customRecipes → app. Safety Rule #1: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`, never change. Safety Rule #2: favorites as ID strings only; `loadFavorites` fetches full data per ID. `STORAGE_KEYS.SEARCH_HISTORY = 'recipe-search-history'`, max 10, deduplicated, locked in. Modal: div overlay with `role="dialog"`, manual focus trapping, Escape + backdrop dismissal. `parseQuantity` handles fractions, `Math.round` for float safety. Compound filter: AND logic, `currentFilters = {category, area}`, roving tabindex. `display: none` to hide non-matching cards, DOM preserved.

Custom recipe CRUD is complete. Safety Rule #3 was established: `CUSTOM_RECIPE_PREFIX = 'custom-'`. All custom recipe IDs must begin with the string `'custom-'`. The reason is collision prevention with TheMealDB IDs. TheMealDB identifies recipes with plain numeric strings (e.g., `"52772"`). If a custom recipe were created with an ID like `"52772"`, the function `getAnyRecipeById` would be unable to determine whether to look in localStorage or call the TheMealDB network endpoint. The `'custom-'` prefix makes the routing decision unambiguous: any ID starting with `CUSTOM_RECIPE_PREFIX` resolves to localStorage; all other IDs resolve to a `lookup.php?i=` network call. `CUSTOM_RECIPE_PREFIX` is defined in `constants.js`.

Custom recipes are stored in localStorage under `STORAGE_KEYS.CUSTOM_RECIPES = 'recipe-custom'`. Unlike the favorites key (which stores only IDs), the custom recipes key stores full Recipe objects — because custom recipes have no backing API from which full data could be fetched on demand. Custom recipe objects conform to the same `{id, title, thumbnail, ...}` schema as TheMealDB recipes, which means the rest of the app (modal rendering, favorites toggle, search results integration) handles custom recipes without any special-casing.

`getAnyRecipeById(id)` is the unified lookup function. It checks `id.startsWith(CUSTOM_RECIPE_PREFIX)`: if true, it reads and searches the `CUSTOM_RECIPES` array from localStorage; if false, it calls `lookup.php?i={id}`. This function is used everywhere an ID needs to resolve to a full Recipe object — modal opening, favorites loading, detail view from the Favorites tab.

The custom recipe form handles both add and edit mode from the same HTML form element. A hidden input field holds the ID of the recipe being edited; it is empty when creating a new recipe. On form submit, if the ID field is empty the handler generates a new ID as `CUSTOM_RECIPE_PREFIX + Date.now()`. If the ID field has a value, the handler finds the matching entry in the `CUSTOM_RECIPES` array and updates it in place. The form is pre-populated with the recipe's current data when opening in edit mode.

Image URL input is validated before accepting. When the user enters a URL and moves focus away from the field, a hidden `<img>` element's `src` is set to the URL. If `onload` fires, the URL is accepted and a green checkmark is shown. If `onerror` fires, the URL is rejected and an inline error message is shown. Only the URL string is stored — image binary data is never fetched or stored.

The star rating uses `<input type="radio">` elements (value 1–5) inside a container with `role="radiogroup"` and an `aria-label`. The stars are visually styled via CSS on the radio labels, but the underlying inputs are the actual form controls. Keyboard users navigate and select ratings using arrow keys via native radio button behavior.

`createRecipeCard` uses `textContent` (not `innerHTML`) for all fields that can contain user-provided data — recipe title, description, ingredient names. If `innerHTML` were used, a custom recipe whose title contained `<script>` tags would execute JavaScript when the card is rendered. `textContent` inserts content as literal text without HTML parsing, preventing XSS.

Export: the full `CUSTOM_RECIPES` array is passed through `JSON.stringify`, wrapped in a `Blob`, and downloaded by programmatically clicking an `<a>` element with an `href` set to `URL.createObjectURL(blob)`. The downloaded file has a `.json` extension. Import: a `<input type="file">` element accepts a file selection. `FileReader.readAsText` reads the file contents. `JSON.parse` deserializes it. Each recipe in the imported array is validated — it must have required fields and its ID must start with `CUSTOM_RECIPE_PREFIX` — before being written to the `CUSTOM_RECIPES` localStorage array.

At the end of S5: search, favorites, detail modal with serving scaler and focus trapping, compound filter with roving tabindex, search history, full custom recipe CRUD (add/edit/delete, image validation, star rating), export/import, XSS-safe card rendering, `getAnyRecipeById` routing. All three Safety Rules documented. Offline cache, performance tuning, accessibility improvements, and deployment are S6.

## Summary6 — T6 Test Context (after S6)
*Input: Summary5 + S6 (~12,000 words) → compressed to ~1,000 words*

A personal recipe collection web app in plain HTML, CSS, and vanilla JavaScript. Zero external dependencies, zero build tooling. Deployed to GitHub Pages via the `/docs` folder on `main` branch. TheMealDB (free, no key, CORS-open) provides all recipe data.

**File load order**: `constants.js` → `mapper.js` → `api.js` → `ui.js` → `favorites.js` → `customRecipes.js` → `app.js`. Each file only uses names from earlier files.

**Recipe schema**: `{id, title, thumbnail, category, area, tags, ingredients [{name, measure}], youtubeUrl, sourceUrl}`. Uniform across TheMealDB recipes and custom recipes.

**Three Safety Rules**:
- Rule #1: `STORAGE_KEYS.FAVORITES = 'recipe-favorites'`. Never change. Renaming orphans all saved favorites.
- Rule #2: Favorites stored as ID string array only — never full Recipe objects. `loadFavorites` resolves each ID with a `lookup.php?i=` call.
- Rule #3: `CUSTOM_RECIPE_PREFIX = 'custom-'`. All custom recipe IDs must begin with `'custom-'` to prevent collision with TheMealDB's plain-integer IDs in `getAnyRecipeById`.

**Persistence — three layers**:
1. *Memory (ephemeral)*: `recipeCache` map (fetched Recipe objects keyed by ID), `currentFilters` object, active view name. Cleared when the tab closes.
2. *localStorage (user data)*: Three keys, all defined in `STORAGE_KEYS` in `constants.js`. `'recipe-favorites'`: ID string array. `'recipe-search-history'`: query string array, max 10, deduplicated. `'recipe-custom'`: full custom Recipe object array. Any new key must be added to `STORAGE_KEYS` before use; keys are permanent once data exists under them.
3. *Cache API (network data)*: Cache namespace `'recipe-collector-v1'`. Caches two endpoints: `categories.php` and `list.php?a=list`. TTL is 24 hours, tracked via a custom `X-Cached-At` response header written at cache time — the server's own `Date` header is unreliable for this purpose because CDN servers may return stale or offset timestamps. `fetchWithCache` opens the cache, checks for a match, validates the `X-Cached-At` timestamp against `CACHE_TTL_MS`, and returns the cached response if fresh. On a miss it fetches from the network, clones the response, appends the `X-Cached-At` header, and puts the modified clone into the cache. A manual refresh button calls `cache.delete()` on both endpoints and re-fetches, providing an escape hatch when the user wants to bypass the TTL. `fetchWithCache` is wrapped in `try/catch` — if the Cache API is unavailable (private browsing mode, storage quota exceeded), the function falls back to a plain `fetch()` call with no caching. Full PWA setup and Service Workers were explicitly rejected: the complexity of install/activate/fetch lifecycle, cache versioning, and background sync was judged disproportionate for a personal single-user tool.

**Key architectural constraints**:
- All modals: `<div>` overlay with `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, manual Tab/Shift+Tab focus trapping. Native `<dialog>` element is banned due to `::backdrop` inconsistency and `returnFocus` behavior differences.
- `parseQuantity`: splits on `/` for fractions, returns non-numeric strings unchanged, applies `Math.round(v * 100) / 100` for float safety.
- Compound filter: AND logic. `currentFilters = {category: string|null, area: string|null}`. Non-matching cards hidden with `display: none` — DOM preserved.
- Filter chips: roving tabindex (one chip has `tabindex="0"`, others `tabindex="-1"`, arrow keys move focus).
- `createRecipeCard`: `textContent` for all user-content fields (prevents XSS from custom recipe data).
- `getAnyRecipeById`: routes `'custom-'` prefix to localStorage, other IDs to `lookup.php?i=`.
- `loading="lazy"` on recipe card images only; hero or above-fold images must not use lazy loading.
- Scripts at bottom of `<body>`.

**Accessibility additions in S6**: `role="tablist"` and `role="tab"` with `aria-selected` on nav buttons. `tabindex="0"` and `role="article"` on recipe cards with event delegation for Enter key. `role="status"` and `aria-live="polite"` on status message div. Visually-hidden `<label>` associated via `for`/`id` with the search input (using `.visually-hidden` CSS class: `position: absolute; width: 1px; height: 1px; overflow: hidden; clip: rect(0,0,0,0)`). `<fieldset>` and `<legend>` wrapping filter chip groups. Skip-to-content anchor link: `position: absolute; top: -100%`, transitions to `top: 0` on `:focus`, jumps to `#recipe-grid-region`.

The project is complete: functional, accessible, performant, deployed.
